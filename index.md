# スクロール表示確認用テストページ

このページは、GitHub Pages + Jekyll で作成したページをスマートフォンやパソコンで表示したときに、文章の読みやすさ、見出しの間隔、箇条書き、表、引用、区切り線などがどのように見えるかを確認するためのテストページです。

実際のお知らせではありません。文章量が多い場合に、どの程度スクロールが必要になるかを確認する目的で作成しています。

## 1. このページの目的

マンション内のお知らせは、掲示板だけでは十分な説明が難しい場合があります。

掲示板には要点だけを掲載し、詳しい説明はQRコード先のWebページに掲載することで、関心のある方が必要な情報を確認できるようになります。

一方で、Webページに長い説明を掲載する場合、次のような点を確認しておく必要があります。

- スマートフォンで読みやすいか
- 見出しが適切に目に入るか
- 文章が詰まりすぎていないか
- 箇条書きが見やすいか
- 表が横にはみ出さないか
- 最後まで読める構成になっているか
- 印刷したときに極端に崩れないか

このページは、それらを確認するための仮の長文です。

## 2. 掲示板とWebページの役割分担

掲示板は、すべての住民に向けて最低限必要な情報を伝える場所です。

たとえば、工事日程、断水のお知らせ、総会開催案内、注意喚起など、住民全員に確実に目に入れてほしい情報は、これまで通り掲示板に掲載する必要があります。

しかし、掲示板にはスペースの制約があります。  
また、長い文章を掲示すると、かえって読まれにくくなることもあります。

そのため、掲示板には概要だけを掲載し、詳しい説明はWebページに掲載する方法が考えられます。

### 掲示板に向いている情報

- 日時
- 場所
- 対象者
- 締切
- 重要な注意事項
- 問い合わせ先
- QRコード

### Webページに向いている情報

- 検討の背景
- 複数案の比較
- 費用の考え方
- よくある質問
- 理事会としての考え方
- 今後の予定
- 補足資料

掲示板を「すべてを説明する場所」とするのではなく、「詳しい説明にたどり着くための入口」として使うことで、情報共有の幅が広がります。

## 3. 掲載する情報の考え方

Webページに情報を載せる場合でも、何でも掲載してよいわけではありません。

GitHub Pagesで公開するページは、基本的にはインターネット上に公開される情報です。  
QRコードを知っている人だけが見るように感じられますが、技術的にはURLを知っている人なら誰でも閲覧できます。

そのため、掲載する情報は慎重に選ぶ必要があります。

### 掲載しやすい情報

- 総会資料の補足説明
- 工事や修繕の一般的な説明
- 検討中の選択肢
- 費用の概算
- 理事会からの一般的なお知らせ
- 生活ルールの案内
- ゴミ出しや共用部利用の注意点
- よくある質問と回答

### 掲載を避けるべき情報

- 個人名
- 部屋番号
- 電話番号
- メールアドレス
- 苦情の詳細
- 住民同士のトラブル内容
- 管理会社との未公開のやり取り
- 防犯上の弱点になる情報
- 内部資料そのもの

Web化は便利ですが、公開情報として扱う前提が必要です。

## 4. 仮の説明文：修繕に関するお知らせ

ここからは、実際のお知らせを想定した仮の文章です。

現在、共用部の一部設備について、経年劣化に伴う修繕の必要性が確認されています。  
理事会では、現時点で把握している状況を整理し、今後の対応方針について検討を進めています。

今回の検討では、単に「修繕が必要かどうか」だけではなく、次の点を重視しています。

- 緊急性がどの程度あるか
- 放置した場合にどのようなリスクがあるか
- 応急処置で対応可能か
- 全体修繕が必要か
- 部分修繕で足りるか
- 費用はどの程度か
- 修繕積立金への影響はどの程度か
- 工事中の住民生活への影響はあるか
- 管理員や理事の運用負担は増えないか

住民の皆さまにとって重要なのは、単に「何をするか」だけではなく、「なぜそれをするのか」「他の選択肢は検討したのか」「費用は妥当なのか」という点だと思います。

そのため、理事会では、できるだけ検討内容を分かりやすく整理したうえで、必要に応じて総会や説明会で共有していくことを考えています。

## 6. よくある質問の表示確認

### Q1. Webページを見られない人はどうすればよいですか？

スマートフォンやパソコンで閲覧できない方のために、必要に応じて紙資料での閲覧方法を用意することが考えられます。

たとえば、管理事務室で閲覧できるようにする、希望者には印刷した資料を渡す、掲示板に要点を掲載する、といった方法があります。

### Q2. Webページの内容は正式な決定ですか？

掲載内容によります。  
検討中の内容であれば、「検討中」と明記する必要があります。

正式な決定事項、理事会での検討状況、総会に向けた補足説明、住民への参考情報は、それぞれ意味が異なります。  
誤解を避けるため、ページ冒頭に位置づけを記載することが重要です。

### Q3. 質問や意見はWebページから送れますか？

最初の段階では、Webページは一方向の情報提供に留めるのが安全です。  
コメント欄を設けると、理事会が常時対応する必要が生じる可能性があります。

質問を受け付ける場合は、回答期限、回答方法、回答対象の範囲を明確にしたうえで、Googleフォームなどを使う方法が考えられます。

## 8. 注意書きのサンプル

以下は、実際のページに入れる可能性がある注意書きの例です。

> 本ページは、理事会からの補足説明を目的としたものです。  
> 総会の議決権行使、委任状、正式な賛否確認を代替するものではありません。  
> 掲載内容は、必要に応じて更新される場合があります。

このような注意書きを入れておくと、Webページの位置づけが明確になります。

## 9. 長文段落の見え方確認

ここでは、あえて少し長めの段落を入れています。

マンション運営では、理事会がすべての情報を詳細に掲示板へ掲載することは現実的ではありません。一方で、情報量が少なすぎると、住民から見ると「なぜその判断になったのか」が分かりにくくなり、結果として不信感につながることがあります。特に、費用が大きい工事や生活に影響する変更については、結論だけではなく、検討の背景、選択肢、費用、今後の進め方を示すことが重要です。

このような情報は、掲示板ではなくWebページに掲載することで、読みたい人が必要なときに確認できます。また、過去の説明を残しておくことで、後から参加した理事や住民にも経緯を共有しやすくなります。情報の蓄積は、理事会の引き継ぎにも役立ちます。

ただし、Webページを作ること自体が目的になってしまうと、かえって運用負担が増える可能性があります。最初は、更新頻度を高くしすぎず、重要なお知らせや補足説明に限定して始めるのが現実的です。

## 10. 箇条書きの見え方確認

以下は、箇条書きの見え方を確認するためのサンプルです。

- 掲示板には概要を掲載する
- 詳細はQRコード先のWebページに掲載する
- スマートフォンで見られない方には紙資料で対応する
- 個人情報は掲載しない
- コメント欄は設けない
- 質問受付は必要に応じて別フォームにする
- ページには作成日と更新日を記載する
- 検討中か決定事項かを明記する

番号付きリストの確認もします。

1. お知らせの概要を作成する
2. 詳細説明をMarkdownで作成する
3. GitHub Pagesで公開する
4. QRコードを作成する
5. 掲示板に掲示する
6. 必要に応じて紙資料を用意する
7. 質問や意見を整理する

## 11. 今後の運用イメージ

今後、Webページを使う場合は、次のような運用が考えられます。

### 掲示板

掲示板には、住民全員に見てほしい要点を掲載します。

- 件名
- 概要
- 対象者
- 締切
- QRコード
- 紙資料の閲覧方法

### Webページ

Webページには、掲示板では書ききれない補足説明を掲載します。

- 背景
- 経緯
- 比較表
- よくある質問
- 今後の予定
- 注意事項

### 紙資料

スマートフォンやパソコンで閲覧できない方向けに、必要に応じて紙資料を用意します。

紙資料を全戸配布にすると負担が大きいため、希望者のみ閲覧または配布する形が現実的です。

## 13. まとめ

このページは、スクロールしたときの見た目を確認するためのテストページです。

実際に公開するページでは、次の点を意識すると読みやすくなります。

- 見出しを多めに入れる
- 1段落を長くしすぎない
- 箇条書きを適度に使う
- 表は必要最小限にする
- 注意書きは冒頭または末尾に置く
- 作成日・更新日を明記する
- 公開してよい内容だけを掲載する

以上でテストページは終了です。
